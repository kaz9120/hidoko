# Autonomous Development — Issue を消化し続けるループ

このスキルは「ユーザーが起こさなくても hidoko の Issue を消化し続ける」ためのもの。`/shipping` の単発実行を前提に、その手前で「次に何をやるか」を決めるレイヤーを担う。

役割は薄い。実装の中身は全部 `/shipping` に任せる。このスキルは Issue の取捨選択とループ運営だけを担当する。

前提:

- `gh` がインストール・認証済み
- 同リポジトリ内の `/shipping` スキル（[.claude/skills/shipping/](../shipping/SKILL.md)）が使える
- ローカルマシンで Claude Code セッションが動いている

---

## 自律性の原則

このループはユーザー不在で回ることを前提にする。`/shipping` 側の「自律性の原則」がそのまま継承される。確認を取るのは新規実装で What/Done が確定できないときだけ。それ以外、特に「どの Issue を次に取るか」「いつループを止めるか」は自分の判断で決める。

`/shipping` 側は取り消し不能アクションも自分でやる方針なので、このループからは「ユーザーに渡す判断」がほぼ発生しない。

---

## ワークフロー

### Step 1: 取り込み候補を取得する

デフォルトのフィルタは次の組み合わせ。引数で上書きできる。

- `state: open`
- ラベル: `ai-ready`（ユーザーが「AI が着手して良い」と明示した Issue）
- Assignee: なし、または自分（PR 作成者）
- Draft PR / 進行中 PR がリンクされていない

```sh
gh issue list \
  --state open \
  --label ai-ready \
  --json number,title,body,labels,assignees,url \
  --limit 20
```

引数でフィルタを指定された場合はそれを優先する。例: `--label feature` だけ取りたい、特定 milestone のものだけ取りたい。

### Step 2: 進行中の PR を先に片付ける

新しい Issue に手を出す前に、自分が着手済みで未完走の PR がないか確認する。

```sh
gh pr list \
  --author "@me" \
  --state open \
  --json number,title,url,isDraft,headRefName,statusCheckRollup
```

未完走の PR（CI が緑になっていない / 未対応コメントがある / Agentation のフィードバックが付いている）があれば、新しい Issue に手を出さず、まずそれを完走させる。これは並列開発の事故を避けるための原則で、後述の「並列処理しない」とつながる。

進行中 PR があるなら、`/shipping <PR_URL>` でその PR を再起動して Step 4 へ。

### Step 3: 次の Issue を選ぶ

進行中 PR がないなら、Step 1 で取得した候補から 1 つ選ぶ。

選択基準（上から順に優先）:

1. ユーザーが直近で言及した Issue があればそれ
2. ラベル `priority:high` が付いているもの
3. ラベル `bug` が付いているもの（バグは溜めない方が良い）
4. milestone に含まれていて締切が近いもの
5. 番号が小さいもの（古いものから消化）

Why / What / Done が `[要確認: ...]` 等で空欄のままなら、`/shipping` 側の What/Done 精緻化フェーズでユーザーに 1 回確認する。ここで自前判断しない。

### Step 4: shipping を起動する

選んだ Issue の URL を引数に `/shipping` を呼ぶ。

```text
/shipping <Issue URL>
```

`/shipping` 側で「PR をマージ・ブランチ削除まで届ける」までを担当する。このループは完了通知を待って Step 5 に進む。

### Step 5: 完走の確認と次サイクルへ

`/shipping` が「マージ完了」と返したら、次のサイクルへ進む。マージ前で止まっている場合（main 由来の CI 失敗待ち等の限定ケース）は、状況を簡潔に記録して次サイクルへ進む（人間判断を待っている間も他の Issue は進められる）。

`ScheduleWakeup` で次のチェックまでスリープする。デフォルトは 30 分（1800 秒）。引数で変えられる。マシンが起きていれば自動で次のサイクルに入る。

---

## 並列処理しない

このループは「1 サイクルに 1 PR」を厳守する。理由は 3 つ。

- 同じ作業ツリーで複数 PR を同時に進めると、共通ファイル（lockfile / 設定 / 共通コンポーネント）でコンフリクトが起きやすい
- ユーザーが 1 つの PR をプレビューで確認している間に別の PR の話を始めると、文脈が混ざる
- セルフレビューと PR コメント対応は逐次のほうが質が出る

明示的に並列開発をしたい場合は、`git worktree add` で別ディレクトリに別作業ツリーを切り、そこで別セッションを立てる。このスキル自体は単一の作業ツリーで完結する前提で動く。

---

## 取り込み Issue 候補がないとき

Step 1 の取得結果が空のとき、いきなりユーザーに振らずに次の順序で粘る。

1. ラベルなしの Open Issue も対象に含めて再検索する
2. それでも空なら、プロダクトの現状・過去 PR の「やらなかったこと」・既知の負債から次の Issue 候補を 3〜5 個提案する形で Issue を起こす（`gh issue create` で `ai-ready` ラベル付き）
3. それも避けるべき状況（プロダクトの方向性が定まっていない、ユーザーが意図的に Issue を空にしている）なら、次サイクルまで `ScheduleWakeup` でスリープして待つ

「ユーザーに止めて良いか確認する」は最後の手段。原則は「ループを止めない」。

---

## ループの止め方

ユーザーが明示的に「止めて」「ループを終わって」と言ったら止める。それ以外は `ScheduleWakeup` で回り続ける。マシンがスリープすれば自動でループも止まり、起きたら次のサイクルに入る。

---

## 重要原則

1. `/shipping` の単発実行を前提にする。実装の中身はそちらに任せる
2. 並列処理しない。1 サイクル 1 PR（worktree で並列したいなら別セッションを立てる）
3. 進行中の PR を先に片付けてから新しい Issue を取る
4. Issue の選択基準を毎回確認しない。並べ替え基準を自分で持つ
5. 候補が空でも粘る（再検索 → 自分で Issue を起こす → 最後にスリープ）
6. 明示の停止指示があるまで止めない
